## 已识别的内在张力（待解）

### 张力 1: Local-First vs 多云同步
- 数据默认在本地
- 同步链路 4 家服务商，每家 API 稳定性/限流/冲突解决不同
- 端到端加密是好设计，但同步层是工程怪兽

### 张力 2: PWA vs 原生
- ADR-0001 已决策走 Capacitor
- 但纯 PWA 仍有零安装优势，trade-off 未完全解

### 张力 3: 11 provider 可选 + 隐私优先 + 本地存储
- 三者本身冲突
- ADR-0003 待决（路径 A 服务端代理 / B SubtleCrypto / C 接受现状）

## 已建立的 Karpathy 风格表达规则

用于撰写 issue 文档 / ADR / 评论时保持风格一致：

- 短句独立成段
- 精确参数 + 口语化强调
- 不用「综上所述」「总结一下」
- 用 imo 标记主观判断
- 「我在这里对 X 真的没有很强的直觉」表达不确定性
- 不 hedge，用确定句式或直接说不知道

## 已砍掉的方案（避免反复重提）

### Jag-5 Confidence Scoring
**原因**：
1. LLM 自我评估是其能力凹陷点（meta-cognition 不可靠）
2. 已有 Jag-1 + Jag-3 更可靠 10 倍
3. 多一条数据通道增加产品复杂度
4. Matt "don't be a hero" 原则：不必要就别加

**何时重提**：仅当 Jag-1 + Jag-3 都失效，且有明确证据需要第三道防线时。

### Jag-6 跨模型交叉验证（推迟到 P1）
**原因**：
1. 成本翻倍
2. 「两路不一致时怎么办」UX 决策未清
3. 需先 prototype 验证设计

### Jag-7 长期记忆修正（推迟到 P2）
**原因**：
1. 涉及数据所有权
2. 涉及隐私边界
3. 涉及长期一致性
4. 需先 grill 后产出 ADR-0004

## 已识别的真洞（不要假装不存在）

1. **API Key 前端注入**：任何 XSS 都能捞。这是 local-first + 多 provider 设计的本质代价。
2. **IndexedDB 不可靠**：iOS 隐私模式、存储压力、用户清缓存都可能丢数据。
3. **维护成本非线性**：11 provider 的差异（流式、工具调用、context window）会让抽象漏掉。
4. **LLM 幻觉**：是 dream machine 的本质属性，不是 bug。产品只能应对，不能消除。

## 已赞的设计选择（不要轻易改）

1. **多通道 Prompt 批量生成**：承认同次召唤会出不同幽灵，对 LLM 本质正确认知
2. **日记/回顾/洞察解耦**：承认 LLM 在不同任务上方差不同
3. **transcribe 静默丢弃**：承认 LLM 在静音输入会幻觉
4. **占用符保护清洗算法**：承认 LLM 输出格式不可靠
5. **任务队列断点续传**：承认网络会挂、手机会锁屏、LLM 会限流
6. **Iron Man 套装定位**：不替用户做决定，让用户拥有控制权

## 后续 session 的工作方式

如果新 session 想继续用 Karpathy 视角评估某模块：
1. 读取该模块相关代码（用 Read 工具，不靠记忆）
2. 按 6 个心智模型（Software X.0 / 构建即理解 / 幽灵 / March of Nines / 锯齿状 / Iron Man）路由
3. 输出一句话判断 + 杠杆最高的 3 个具体建议
4. 不堆名言、不重复之前会话的判断
5. 重大发现更新到本文件
