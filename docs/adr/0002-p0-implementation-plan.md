# ADR-0002: P0 实施计划与 8 issue 排序

**状态**：已决策（2026-07-19）
**驱动**：Karpathy 视角对项目的工程现实主义与锯齿状智能两项评估
**执行计划**：`docs/issues/p0/`

## 背景

2026-07-19 Karpathy 视角评估后，两项关键指标被判定为「中等偏上」而非「极强」：
- 工程现实主义（March of Nines）
- AI 能力边界处理（Jagged Intelligence）

差距分析见 Karpathy session log。结论是需要 P0 级别的工程加固。

## 决策

### 8 个 P0 issue

**March of Nines 类**（工程现实主义）：
1. `#001` 版本号与发布纪律
2. `#002` 服务端超时 + 降级
3. `#003` 任务队列指数退避
4. `#006` 错误日志环形缓冲
5. `#007` 存储预警
6. `#008` 自动备份

**Jagged Intelligence 类**（AI 能力边界处理）：
1. `#004` 转写幻觉检测升级
2. `#005` 引用回溯验证

### 排序逻辑

```
#001 → #002 → #003 → #004 → #005 → #006 → #007 → #008
```

**硬约束**：
- `#004` 和 `#008` 修改 db schema（v14、v15），必须顺序执行以避免迁移冲突
- `#005` 不依赖任何前置，但放在 `#004` 之后渲染约定清晰
- `#001` 最先做，建立发布纪律信号

### 为什么是这 8 个

每个候选评估三个标准：
1. 杠杆（解决多少真问题）
2. 工作量（小/中/大）
3. 风险（低/中/高）

只有「杠杆高 + 工作量小 + 风险低」或「杠杆极高 + 风险可控」才进 P0。

### 为什么砍掉 Jag-5（Confidence Scoring）

理由（来源：Karpathy 评估）：
1. LLM 自我评估是其能力凹陷点（meta-cognition 不可靠）
2. 已有 Jag-1（引用回溯）+ Jag-3（用户反馈）更可靠 10 倍
3. 多一条数据通道反而增加产品复杂度
4. Matt "don't be a hero" 原则：不必要就别加

## 推论

### 接受的风险
- 8 个 issue 中 `#004`、`#008` 涉及 schema 迁移，需要充分回归测试
- `#005` 性能开销需要监控（一篇长日记几千引用的情况）
- `#008` 不备份 attachments（音频 Blob 太大），文档必须明示

### 推迟的理由
- **P2-9 API Key 真隔离**：local-first + 11 provider + 隐私优先三者本身冲突，需 grill 后产出 ADR-0003
- **Jag-6 跨模型交叉验证**：成本翻倍，需先 prototype 验证 UX
- **Jag-7 长期记忆修正**：涉及数据所有权、隐私边界、长期一致性，需 grill 后产出 ADR-0004

## 后续

- 每个 issue 完成后立即合并 main + 打 tag（如果是 minor）
- 全部完成后做一次端到端回归
- 启动 P1 时先 grill 一次，避免 P0 期间沉淀的新约束被忽略
