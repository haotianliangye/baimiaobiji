# 白描笔记 V2 重构 PRD：从语音日记到个人数据枢纽

## Problem Statement

白描笔记最初是一个「语音拾微 → AI 日记/回顾/洞察」的单一流程工具。随着使用深入，用户发现：

1. **板块概念重叠**：「日记」和「回顾」都是对拾微的 AI 整理，功能交叉，用户认知负担重。
2. **缺少慢思考载体**：只有快记录（拾微）和 AI 产出，缺少一个用户主动沉淀深度思考的笔记空间。
3. **AI 能力分散**：语义搜索（Copilot）只用于检索，缺少一个随手可用的通用 LLM Chat 窗口。
4. **Prompt 交互陈旧**：当前 Prompt 选择缺少选中态、自定义名称不可改、生成按钮与多选逻辑不匹配。
5. **数据维度单一**：拾微以文本和音频为主，缺少图片、视频、链接等多维原始输入。
6. **个人信息被多个 App 割裂**：行为（拾微）、事物（多媒体）、思考（笔记/对话）分散在不同应用，无法形成立体个人数据。
7. **多语言与可访问性不足**：应用仅中文，且缺少 TTS 朗读能力，不利于复习与无障碍使用。

用户希望把白描笔记升级为一个**隐私优先、本地优先、AI 辅助的个人数据枢纽**：快思考有拾微，慢思考有沉淀，AI 整理有回顾，模式洞察有明悟，日常对话有 LLM Chat，所有数据留在本地、统一检索、统一导出。

## Solution

将白描笔记的信息架构从「记录 / 日记 / 回顾 / 洞察 / Copilot」重构为「拾微 / 回顾 / 沉淀 / 明悟」，并围绕这个核心四象限补齐能力：

- **拾微**：保留快系统输入，支持语音、文字、图片、音频、视频、链接；语音走 STT，多媒体保留原始文件并生成文本摘要。
- **回顾**：合并原「日记」与「回顾」，作为 AI 对拾微的整理产物。Prompt 配置合并为「日记回顾生成 Prompt」，提供 5 个可选槽位（日记 / 回顾 / 自定义 1/2/3），支持多选、自定义改名、中英独立保存。
- **沉淀**：新增 flomo 式笔记系统，支持 Markdown、标签、附件；默认瀑布流展示，可切时间线；是用户慢思考的沉淀空间。
- **明悟**：由原「洞察」升级而来，包含「明悟」与「洞察」两类 AI 产出，手动按日 / 周 / 月 / 季度 / 半年生成，数据源为拾微 + 沉淀。
- **LLM Chat**：在现有 Copilot 面板中新增通用对话模式，与 RAG 问答并列；聊天记录独立保存、可导入导出，不参与明悟/洞察数据源。
- **TTS**：为回顾、明悟、洞察、LLM Chat 的 AI 产出提供一键朗读，支持浏览器原生 TTS 与外部 API 双服务。
- **随机漫步**：从全部个人数据中随机抽取 3 条近期未展示过的记录，以卡片形式滑动浏览，支持已阅 / 标签 / 编辑 / 复制 / 删除。
- **标签管理**：全局共享标签体系，支持扁平与层级标签；AI 产出自动打标，用户原创内容手动打标；设置中提供标签树管理页。
- **数据管理**：统一导入 / 导出 / 时间范围选择，覆盖聊天记录、拾微、回顾、沉淀、明悟。
- **多语言**：UI 与 AI Prompt 均支持中 / 英切换，内容支持中英混合输入与跨语言语义检索。

## User Stories

### 信息架构与导航

1. 作为用户，我希望底部导航是「拾微 / 回顾 / 沉淀 / 明悟」，这样四个板块概念清晰、互不重叠。
2. 作为老用户，我希望应用启动后自动把旧「日记」和「回顾」合并、把「洞察」升级为「明悟」，这样我的历史数据不会丢失。
3. 作为用户，我希望 Copilot / LLM Chat 仍从顶部 Header 的图标进入，这样不占用底部 Tab。

### 拾微输入

4. 作为用户，我希望在拾微输入框中直接输入文字或语音，这样能快速捕捉想法。
5. 作为用户，我希望在拾微中附加图片、音频、视频、链接，这样记录更立体。
6. 作为用户，我希望语音拾微自动转写成原文（STT），而不是摘要，这样保留原始表达。
7. 作为用户，我希望图片 / 视频拾微自动生成文字摘要，这样后续检索和生成回顾时能用到。
8. 作为用户，我希望多媒体原始文件本地保留且不压缩，这样未来模型能力更强时可以复用。

### 回顾生成

9. 作为用户，我希望在回顾页面点击「AI 智能整理」后，弹出的 Prompt 选择器里有「日记 / 回顾 / 自定义 1/2/3」五个选项，这样我知道有哪些视角可生成。
10. 作为用户，我希望点击某个 Prompt 选项时左侧出现对勾表示选中，再次点击取消，这样选择状态明确。
11. 作为用户，我希望至少保留一个 Prompt 被选中，这样不会误操作导致全部取消。
12. 作为用户，我希望选中的多个 Prompt 各生成一篇独立的回顾卡片，这样我可以对比不同视角。
13. 作为用户，我希望自定义 Prompt 的名字可以改成「知识」「决策」等，这样生成面板上显示的是我理解的名称。
14. 作为用户，我希望默认自动选中「日记 + 回顾」，这样每天自动生成这两个视角，同时我也可以手动增选自定义 Prompt。
15. 作为用户，我希望设置里能选择「生成回顾时是否向模型提交多媒体」，这样非多模态模型不会报错。

### 沉淀笔记

16. 作为用户，我希望沉淀是一个独立的笔记板块，支持 Markdown 文本、标签、附件，这样我能沉淀深度思考。
17. 作为用户，我希望沉淀默认按瀑布流展示，同时也能切换成按日期分组，这样符合不同浏览习惯。
18. 作为用户，我希望双击一条沉淀记录就进入编辑，这样修改成本低。
19. 作为用户，我希望沉淀的输入框展开后参考 Blinko 的富文本编辑器，有格式工具栏、标签、附件入口，这样输入体验好。
20. 作为用户，我希望沉淀支持修改创建时间但保留初始时间，这样我可以补记过去的思考。
21. 作为用户，我希望沉淀使用全局共享的标签体系，这样拾微、沉淀、回顾、明悟可以用同一套标签串联。

### 明悟与洞察

22. 作为用户，我希望明悟板块包含「明悟」和「洞察」两类 AI 产出，这样既能看透本质也能看到数据模式。
23. 作为用户，我希望明悟/洞察沿用 5 槽 Prompt 模型（默认 + 自定义 1/2/3），但不自动生成，这样我完全控制何时生成。
24. 作为用户，我希望明悟/洞察的数据源同时包含拾微和沉淀，这样慢思考的内容也能进入模式分析。
25. 作为用户，我希望生成明悟/洞察时可以选择时间范围（当日 / 周 / 月 / 季度 / 半年），这样我能做不同粒度的反思。

### LLM Chat

26. 作为用户，我希望在 Copilot 面板中有一个通用 LLM Chat 模式，这样不用跳去其他 App 就能和 AI 聊天。
27. 作为用户，我希望 LLM Chat 的聊天记录保存在本地，并自动生成会话标题，这样我能回顾历史。
28. 作为用户，我希望 LLM Chat 聊天记录支持导入 / 导出，这样数据可迁移。
29. 作为用户，我希望 LLM Chat 聊天记录不参与明悟/洞察的数据源，这样聊天内容不会污染个人分析。

### 标签系统

30. 作为用户，我希望用户原创的拾微和沉淀手动打标签，这样标签反映我的分类意图。
31. 作为用户，我希望 AI 生成的回顾和明悟/洞察自动打标签，这样减少整理成本。
32. 作为用户，我希望标签支持层级（如 `#工作/项目A`），并且搜索父标签能显示子标签内容，这样分类更灵活。
33. 作为用户，我希望设置里有一个标签管理页，可以重命名、合并、删除标签，这样长期积累后标签不会混乱。
34. 作为用户，我希望合并标签后未来再输入被合并的标签名会自动纠正为合并后的标签，这样避免重复标签。

### TTS

35. 作为用户，我希望在回顾、明悟、洞察、LLM Chat 的 AI 产出旁有一个播放按钮，这样我能听内容。
36. 作为用户，我希望 TTS 支持浏览器原生朗读和外部 API 两种服务，这样有网无网都能用。
37. 作为用户，我不希望拾微和沉淀自动朗读，因为这些内容可能太零散或私密。

### 随机漫步

38. 作为用户，我希望点击沉淀板块右上角的灯泡图标进入随机漫步，随机看到过去 3 条记录，这样能重温旧想法。
39. 作为用户，我希望随机漫步默认从沉淀和回顾中抽取，但也可以扩展为全部数据，这样范围可控。
40. 作为用户，我希望近期已经展示过的记录不会重复出现，这样每次都有新鲜感。
41. 作为用户，我希望随机漫步的卡片可以标记已阅、编辑标签、编辑内容、复制、删除，这样能快速处理。

### 多语言

42. 作为用户，我希望在设置中切换 UI 语言为中文或英文，这样界面语言符合我的习惯。
43. 作为用户，我希望切换语言后 AI Prompt 也自动切换到对应语言版本，这样 AI 输出与界面语言一致。
44. 作为用户，我希望可以用中英混合输入拾微和沉淀，并且语义搜索能跨语言匹配，这样输入不受语言限制。

### 数据管理与迁移

45. 作为老用户，我希望应用升级后自动迁移旧数据并先备份，这样我不会丢失数据。
46. 作为用户，我希望数据管理页面支持按时间范围导入 / 导出拾微、回顾、沉淀、明悟、聊天记录，这样数据完全可控。
47. 作为用户，我希望迁移完成后旧表被删除，避免占用 IndexedDB 空间。

### 搜索与分块

48. 作为用户，我希望所有文本内容的分块逻辑统一，这样语义检索结果更准确。
49. 作为用户，我希望拾微的多媒体内容既用文本摘要做 Embedding，也对多媒体本身做索引，这样未来多模态搜索更完整。
50. 作为用户，我希望回顾 / 明悟 / 洞察的 AI 长文本按原文分块，而不是只索引摘要，这样搜索能命中正文细节。

## Implementation Decisions

### 1. 核心信息架构

- 底部导航固定为 4 个 Tab：**拾微**、**回顾**、**沉淀**、**明悟**。
- 原 `/diary` 与 `/review` 合并为 `/review`；原 `/insights` 改为 `/mingwu`（路由与组件语义同步）。
- Copilot / LLM Chat 保持为全局覆盖层，从 Header 的 MessageSquare 图标进入，不新增底部 Tab。
- 路由映射：
  - `/` → 拾微（原 Record）
  - `/review` → 回顾（合并原 Diary + Review）
  - `/thoughts` → 沉淀（新增）
  - `/mingwu` → 明悟（原 Insights）

### 2. 数据库 Schema 变更

新增与修改的表：

- `raw_logs`：保持不变，继续存储拾微。
- `daily_reviews`：扩展为合并后的回顾表。原 `daily_diaries` 的数据迁移进来，保留 `prompt_index` / `prompt_name` 字段以区分「日记」「回顾」视角。
- `thoughts`：新增表，字段包括 `id`、`content`（Markdown）、`tags`（数组）、`attachments`（Blob / URL 元数据）、`created_at`、`original_created_at`（支持修改创建时间）。
- `mingwu`：由 `insights` 改名而来，字段基本不变，新增 `mingwu_type`（`'mingwu' | 'insight'`）和 `prompt_index` / `prompt_name`。
- `copilot_conversations`：保持不变，新增 `mode` 字段以区分 RAG 问答与通用 Chat。

迁移策略：

- 应用启动时自动检测旧表；若存在则先导出备份，再执行迁移。
- `daily_diaries` 中的记录按 `prompt_name = '日记'` 合并进 `daily_reviews`。
- `insights` 表改名并扩展字段；迁移完成后删除 `daily_diaries` 与旧 `insights` 表。

### 3. Prompt 配置模型重构

设置中的 Prompt 配置从「日记 / 回顾 / 洞察 三列独立」改为：

- **日记回顾生成 Prompt**：5 个槽位
  - `日记`（默认，不可改名）
  - `回顾`（默认，不可改名）
  - `自定义 1`、`自定义 2`、`自定义 3`（可改名，按区域独立）
- **明悟生成 Prompt**：5 个槽位（默认「明悟」+ 自定义 1/2/3）
- **洞察生成 Prompt**：5 个槽位（默认「洞察」+ 自定义 1/2/3）

每个 Prompt 区域维护独立的「选中状态」集合，默认选中前两个默认槽位（对回顾而言是「日记 + 回顾」）。选中状态持久化在设置 store 中，手动生成与自动生成共用同一状态。

语言维度：

- 每种语言独立保存一套 Prompt 配置。
- 切换 UI 语言时，Prompt 配置与默认 Prompt 文案同步切换。

### 4. 生成面板交互

- 沿用现有浮层菜单，但改造为多选：
  - 每个 Prompt 项左侧显示 checkbox，选中为对勾。
  - 右侧小字显示「已配置」状态（可选）。
  - 点击项切换选中态；只剩一项时不可取消，必须先选中另一项。
- 顶部「全部生成」按钮文案改为「生成 N 篇回顾」，N 为当前选中 Prompt 数。
- 生成中：浮层关闭，按钮进入 loading，顶部 header 显示队列状态。
- 生成结果：按 `prompt_name` 在卡片 sub-header 上区分「日记」「回顾」「知识」等视角。

### 5. 自动生成队列适配

- 复用现有基于 `localStorage` 的任务队列：`autoGenTasks`。
- 任务模型从 `type: 'diary' | 'review'` 改为 `type: 'review'`（因为已合并）。
- 任务扫描逻辑改为检查「当前选中的 Prompt 索引」与 `daily_reviews` 中已存在的 `prompt_index` 差集。
- 每个任务对应一个（日期, promptIndex）组合，串行执行、失败重试 3 次、API 限流间隔保留。

### 6. 沉淀模块

- 数据单元：一段 Markdown 文本 + 标签数组 + 附件数组 + 创建时间 + 初始创建时间。
- 列表视图：默认瀑布流，顶部提供「瀑布流 / 时间线」切换。
- 输入：底部快速输入框，点击展开为 Blinko 风格的富文本编辑器（格式工具栏、标签入口、附件入口）。
- 编辑：双击记录进入编辑页 / 弹窗。
- 时间：支持修改创建时间；排序与分组以修改后的创建时间为准，但保留 initial_created_at 用于溯源。
- 附件：图片 / 音频 / 视频 / 链接；原始文件存 IndexedDB Blob；为明悟/洞察生成时直接提交文本与附件。

### 7. 标签系统

- 标签字符串支持 `parent/child` 层级语法；数据库中存储为完整路径字符串。
- 搜索父标签时，通过前缀匹配同时返回子标签内容。
- 标签来源：
  - 拾微、沉淀：用户手动输入 `#标签` 或从标签建议中选择。
  - 回顾、明悟、洞察：AI 生成内容后异步调用标签生成 Prompt，用户可在卡片上删改。
- 标签管理页：
  - 树形展示所有标签（扁平标签在根级）。
  - 支持重命名（级联更新所有记录）、合并（建立别名映射）、删除（解除关联）。
  - 合并后，用户再次输入被合并标签名时自动纠正为合并目标。

### 8. 明悟 / 洞察模块

- 页面继承原 Insights 的时间维度：当日 / 周 / 月 / 季度 / 半年。
- Prompt 配置区分为「明悟生成 Prompt」与「洞察生成 Prompt」，各 5 槽。
- 数据源：所选时间范围内的 `raw_logs` + `thoughts`。
- 生成时若使用多模态模型，则同时提交文本与多媒体附件；否则仅提交文本摘要。
- 卡片 sub-header 区分「明悟」与「洞察」。

### 9. LLM Chat

- 在 Copilot 面板中新增模式切换：「RAG 问答」/「通用 Chat」。
- 通用 Chat 不走 `retrieveCopilotContext`，仅上传 messages 到 `/api/chat`。
- 会话表 `copilot_conversations` 增加 `mode` 字段；标题从首条用户消息自动截取前 20 字。
- 历史会话页支持单条删除与单条导出（Markdown）。
- 设置 → 数据管理支持批量导入 / 导出所有聊天记录（Markdown + JSON），支持时间范围筛选。

### 10. TTS

- 覆盖范围：回顾、明悟、洞察的 AI 产出，以及 LLM Chat 的 AI 回复。
- 每个内容块旁放置播放按钮；点击后开始朗读，再次点击停止。
- 设置中选择 TTS 服务：浏览器 Web Speech API（默认）或外部 TTS API（如豆包 seed-tts）。
- 语言跟随当前内容语言检测，或允许用户指定默认朗读语言。

### 11. 随机漫步

- 入口：沉淀板块右上角灯泡图标。
- 默认数据源：`thoughts` + `daily_reviews`；设置中可扩展为 `raw_logs` + `thoughts` + `daily_reviews` + `mingwu`。
- 每次随机抽取 3 条，过滤掉近期（如最近 7 天）已展示过的记录；记录展示历史存 `localStorage`。
- 展示形态：卡片堆叠滑动（参考 Blinko），底部操作栏：已阅 / 标签 / 编辑 / 复制 / 删除。

### 12. 数据分块与向量化对齐

- 统一分块 pipeline：所有文本内容进入向量索引前经过同一套清洗 → 分块 → Embedding → 存储流程。
- 分块粒度统一：基于 token 数（建议使用 `js-tiktoken` 或自研轻量分词器），避免引入 LangChain/llamaindex 增大包体积。
- 元数据统一：每个 chunk 携带 `source_type`（record / thought / review / mingwu）、`source_id`、`created_at`、`tags`。
- 多媒体索引：
  - 图片 / 视频：生成文本摘要后对该摘要做 Embedding。
  - 同时保留多媒体文件引用，为未来多模态 Embedding 做准备。

### 13. 多语言

- UI 文案使用 i18n 字典（中 / 英）。
- AI Prompt 按语言独立保存；切换语言时同时切换默认 Prompt。
- 内容语言不强制；语义搜索使用 Gemini Embedding 的多语言能力实现跨语言匹配。

### 14. 数据管理

- 设置 → 数据管理增加：
  - 导出：选择时间范围、选择数据类型、选择格式（Markdown / JSON）。
  - 导入：上传 JSON，按类型写入对应表；冲突策略为「以导入为准」或「跳过」。
  - 迁移备份：自动导出最近一次迁移的备份文件。

## Testing Decisions

### 测试策略

项目目前没有单元测试，但 `devDependencies` 已包含 `puppeteer`。本次重构采用**端到端浏览器测试**作为主要测试缝（最高层 seam），覆盖关键用户旅程，避免在大量 UI 重构期间建立脆弱的单元测试网。

理想情况下只维护一套 E2E 测试；若后续出现复杂纯函数（如分块算法、标签层级解析），再补充少量单元测试。

### 测试范围

使用 Puppeteer 在本地启动应用后验证：

1. **数据迁移旅程**：构造旧版 IndexedDB（含 `daily_diaries` 与 `insights`），启动应用，验证数据出现在新表且旧表被删除。
2. **回顾生成旅程**：在拾微页创建一条记录 → 进入回顾页 → 选择「日记 + 知识」两个 Prompt → 点击生成 → 验证出现两篇独立卡片，sub-header 显示正确名称。
3. **Prompt 多选边界**：在生成面板取消最后一个 Prompt，验证无法取消；切换到另一个后再取消原项，验证状态正确。
4. **沉淀 CRUD 旅程**：在沉淀页创建一条带标签的笔记 → 验证瀑布流展示 → 双击编辑 → 修改创建时间 → 验证时间线视图正确分组。
5. **标签管理旅程**：创建两条带 `#工作/A` 和 `#工作/B` 的沉淀 → 进入标签管理合并 `#工作/A` 到 `#工作/B` → 搜索 `#工作` 验证两条都出现。
6. **明悟生成旅程**：创建拾微和沉淀 → 进入明悟页选择「本周」与「明悟」Prompt → 生成后验证卡片存在且包含两类数据源的信息。
7. **LLM Chat 旅程**：打开 Copilot → 切换到通用 Chat → 发送消息 → 验证历史会话列表出现该会话且标题正确。
8. **TTS 旅程**：生成一篇回顾 → 点击播放按钮 → 验证 Web Speech API 被调用（通过 mock 或 console 事件）。
9. **随机漫步旅程**：创建 5 条沉淀 → 进入随机漫步 → 验证展示 3 条 → 点击已阅后刷新不再展示该条。
10. **多语言切换旅程**：切换 UI 为英文 → 验证生成面板 Prompt 名称变为英文默认；切换回中文验证恢复。

### 什么算好的测试

- 只测试外部可观察行为（页面文案、卡片数量、数据库内容、网络请求），不测试内部状态或具体组件实现。
- 每个测试在独立会话中运行，测试结束后清理 IndexedDB 与 localStorage。
- 优先使用 `data-testid` 或稳定文案定位元素，避免依赖 CSS 类名。

## Out of Scope

1. **原生 App / Capacitor 迁移**：本次仍基于 PWA，端侧 AI 相关预研另见 `docs/on-device-ai-migration.md`，不进入本 PRD。
2. **端侧模型运行**：ASR / Embedding / Chat 的端侧化不在本次范围。
3. **社交与分享**：不支持公开分享笔记、协作、评论。
4. **实时同步冲突解决 UI 增强**：云同步保持现有行为，不新增冲突可视化。
5. **手写 / 绘图输入**：多媒体输入暂不支持手写笔迹。
6. **AI 主动推荐**：除了自动打标签和随机漫步，不增加 AI 主动推送建议的功能。
7. **视频在线播放优化**：视频附件本地保存，播放使用原生 `<video>`，不做特殊流媒体优化。

## Further Notes

- **隐私与本地优先**：所有用户原创内容、AI 产出、聊天记录默认仅存储在本地 IndexedDB；导出文件由用户自行保管。
- **PWA 限制**：IndexedDB 容量受浏览器配额管理，建议在设置中引导用户安装 PWA 以获得持久存储权限；大视频文件需提示用户注意容量。
- **模型能力假设**：多模态输入（图片 / 视频 / 音频混排）默认面向 Gemini 等多模态模型；若用户使用非多模态模型，应开启「生成时不提交多媒体」设置。
- **向后兼容**：Settings store 的版本号需要升级，以正确迁移旧 Prompt 配置到新结构。
- **性能注意**：自动生成队列改为 5 槽后，若用户选中全部 5 个 Prompt，每天最多生成 5 篇回顾，需确保队列 UI 与限流逻辑能清晰反馈进度。
- **随机漫步的「近期不重复」**：建议以会话级或 7 天滑动窗口记录已展示 ID，避免长期历史被反复展示；可配置。
